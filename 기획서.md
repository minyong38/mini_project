# SharedScheduleManager 프로젝트 기획서 및 결과 보고서

> **프로젝트명** : SharedScheduleManager (공유 일정 관리자)
> **개발 형태** : Qt 기반 C++ 데스크탑 애플리케이션 (클라이언트-서버 구조)
> **개발 기간** : YYYY-MM-DD ~ YYYY-MM-DD
> **팀 구성** : ○○○, ○○○, ○○○

## 1. 프로젝트 개요

### 1-1. 한 줄 소개
**"일정을 공유하고, 그 자리에서 바로 대화할 수 있는 통합 협업 메신저"**
개인 일정 관리 + 팀/친구 단위의 공유 캘린더 + 실시간 채팅(단체·1:1·공유캘린더 전용) 을 하나의 네이티브 데스크탑 앱에서 제공합니다.

### 1-2. 기획 의도 (처음 정한 방향)
우리는 프로젝트 시작 시점에 다음과 같은 문제의식을 가졌습니다.

- 팀플·동아리·가족 일정은 주로 **Google Calendar** 같은 일정 앱에서 관리하지만,
- 그 일정에 대한 대화는 **카카오톡 / Discord** 같은 메신저에서 따로 진행된다.
- 그 결과 "이 일정에 대해 논의한 맥락"이 스크롤 저 아래로 묻히고, 일정을 본 사람과 대화를 본 사람이 어긋난다.

→ 우리는 **"일정 = 대화방 단위"** 라는 컨셉으로, 하나의 공유 캘린더 안에서 일정과 대화가 공존하는 경량 데스크탑 앱을 만들기로 결정했습니다.

### 1-3. 최종 결과 요약
초기 기획대로 **일정–대화–참여자가 한 공간에서 유기적으로 연결되는 앱**을 완성했으며, 여기에 더해 기획 단계에서 예정하지 않았던 **프로필 사진 / 닉네임 / 날씨 위젯 / 다크모드 / 시스템 트레이 / 자동 재접속** 등 사용성 개선 기능을 추가 구현하였습니다.

| 항목 | 내용 |
|---|---|
| 총 코드 라인 | 약 **7,370 LOC** (cpp/h 합산) |
| 언어 / 표준 | C++17 |
| 프레임워크 | Qt 6.11 (Widgets, Network, Sql, Multimedia) |
| 빌드 시스템 | CMake 3.16+ / MinGW 64-bit |
| DB | SQLite (서버 내장) |
| 네트워크 | 자체 설계한 텍스트 기반 TCP 프로토콜 (포트 25000) |
| 외부 연동 | Google OAuth 2.0, 기상청 Weather API |

### 2-2. 핵심 사용 시나리오
1. 사용자 A가 로그인하면, 친구 목록에 온라인 상태가 **초록 점**으로 즉시 반영된다.
2. A가 공유 캘린더 "동아리"를 만들고 B, C를 초대한다. → B, C의 화면에 새 탭이 즉시 생성된다.
3. A가 4월 25일에 "정기 회의"라는 일정을 추가한다. → `SHUPDATE` 브로드캐스트로 B, C 화면이 즉시 갱신된다.
4. B가 그 일정 옆 채팅 버튼을 눌러 "그날 저 30분 늦어요"라고 메시지를 남긴다. → 일정과 대화가 한 맥락에 묶인다.

## 3. 시스템 아키텍처

### 3-1. 전체 구조
```
        ┌─────────────────┐                    ┌─────────────────┐
        │   Client (A)    │                    │   Client (B)    │
        │  [Qt Widgets]   │                    │  [Qt Widgets]   │
        └────────┬────────┘                    └────────┬────────┘
                 │                                      │
                 │      TCP (Port 25000, 자체 프로토콜)   │
                 │                                      │
                 └─────────────────┬────────────────────┘
                                   ▼
                        ┌───────────────────────┐
                        │  ScheduleServer       │
                        │  (QTcpServer)         │
                        │  - 메시지 라우팅        │
                        │  - 온라인 상태 관리     │
                        │  - 브로드캐스트          │
                        └──────────┬────────────┘
                                   │
                                   ▼
                        ┌───────────────────────┐
                        │  SQLite (schedule.db) │
                        │  10개 테이블           │
                        └───────────────────────┘
```

### 3-2. 클라이언트 내부 구조
```
MainWindow (메인 화면, 탭 기반)
  ├── CustomCalendarWidget  (개인 / 공유 / 친구 캘린더 - 공통 위젯)
  ├── ScheduleDialog        (일정 추가·수정·삭제 모달)
  ├── ChatDialog            (단체/DM/공유 채팅 공통 다이얼로그)
  ├── SharedCalDialog       (공유 캘린더 생성·관리)
  ├── MyPageDialog          (프로필 사진·닉네임 변경)
  ├── LoginDialog / RegisterDialog
  ├── WeatherManager        (기상청 API 호출)
  └── GoogleAuthManager     (OAuth 2.0 로그인)
```

### 3-3. 서버 내부 구조
- `QTcpServer` 상속 → `onNewConnection()` 에서 클라이언트 수락
- 소켓별 **수신 버퍼** 를 `QMap<QTcpSocket*, QString>` 로 관리 (프레임 분할 대응)
- `\n` 단위로 메시지를 나눠 `handleRequest()` 에서 명령어 파싱
- 명령에 따라 **DB 쿼리 → 응답** 또는 **관련 클라이언트에게 브로드캐스트**
- 읽음 표시는 **메모리에만 저장** (`m_unreadCount`, `m_lastReadRowid`) 하여 오버헤드 최소화

## 5. 통신 프로토콜 설계

### 5-1. 설계 원칙
- **텍스트 기반** — 디버깅 편의성 / Wireshark 로 즉시 확인 가능
- **라인(`\n`) 단위 프레이밍** — 부분 수신 시 `\n` 이 올 때까지 버퍼링
- **`:` 구분자** — 명령어와 인자, 인자들 사이를 구분
- **메시지 안의 `:` 문자** — 마지막 필드로 몰아 `join(":")` 로 복원

### 5-2. 명령어 카테고리 (총 30+ 종)

| 분류 | 대표 명령 | 설명 |
|---|---|---|
| 로그인 | `LOGIN` / `LOGIN_OK` / `LOGIN_REJECT` / `LOGIN_FAIL` | 중복 로그인·비번 불일치 구분 |
| 회원가입 | `SIGNUP` / `SIGNUP_OK` / `SIGNUP_FAIL` | |
| 온라인 | `ONLINE` / `REQUSERS` / `RESUSERS` | 접속 알림 / 유저 목록 |
| 개인 일정 | `ADD` / `DEL` / `MOD` / `REQ` / `RES` / `REQMONTH` / `RESMONTH` | 일자·월 단위 조회 |
| 프로필 | `PROFILE_UPLOAD` / `PROFILE_REQ` / `PROFILE_RES` / `PROFILE_DELETE` | Base64 인코딩 바이너리 |
| 닉네임 | `NICK_UPDATE` / `NICK_REQ` / `NICK_RES` | |
| 단체 채팅 | `CHAT` / `CHATRES` / `REQCHAT` / `RESCHAT` / `READRES` | 읽음 카운트 포함 |
| 1:1 DM | `DM` / `DMRES` / `REQDM` / `RESDM` | |
| 공유 캘린더 | `CREATECAL` / `DELCAL` / `REQCALS` / `RESCALS` / `CALID` / `CALREMOVED` | |
| 공유 일정 | `ADDSH` / `DELSH` / `MODSH` / `REQSHMONTH` / `RESSHMONTH` / `SHUPDATE` | 브로드캐스트 |
| 공유 채팅 | `SHCHAT` / `SHCHATRES` / `REQSHCHAT` / `RESSHCHAT` | |

### 5-3. 예시 흐름 — 공유 일정 추가
```
[Client A]  ADDSH:3:alice:2026-04-25:정기회의 ↵
     ↓
[Server]    (DB INSERT) → (캘린더 3 멤버 전원에게)
     ↓
[Client A,B,C]  SHUPDATE:3 ↵      ← 전원 수신 → 해당 달 재요청
[Client A,B,C]  REQSHMONTH:3:2026-04 ↵
[Server]        RESSHMONTH:3:2026-04-25@17@정기회의|... ↵
```

## 6. 데이터베이스 설계

SQLite 단일 파일(`schedule.db`), 총 **10 테이블**.

| 테이블 | 주요 컬럼 | 목적 |
|---|---|---|
| `users` | USER_ID (PK), PASSWORD | 계정 |
| `profiles` | USER_ID (PK), PHOTO (Base64) | 프로필 사진 |
| `nicknames` | USER_ID (PK), NICKNAME | 닉네임 |
| `schedules` | USER_ID, DATE, CONTENT | 개인 일정 |
| `chats` | USER_ID, MESSAGE, SEND_TIME | 단체 채팅 |
| `dm_chats` | SENDER, RECEIVER, MESSAGE, SEND_TIME | 1:1 DM |
| `shared_calendars` | ID (PK, AUTO), OWNER, NAME | 공유 캘린더 |
| `shared_members` | CAL_ID, USER_ID | 공유 캘린더 멤버 (N:M) |
| `shared_schedules` | CAL_ID, USER_ID, DATE, CONTENT | 공유 일정 |
| `shared_chats` | CAL_ID, SENDER, MESSAGE, SEND_TIME | 공유 채팅 |

> **설계 메모** : 프로필 사진을 별도 파일이 아닌 DB 에 Base64 로 저장한 것은, **서버-클라이언트 간 파일 전송 프로토콜을 별도 설계하지 않고** 기존 텍스트 프로토콜을 그대로 재사용하기 위한 결정이었습니다. 대신 클라이언트 측 캐싱(`m_profileCache`) 을 도입해 재요청을 최소화했습니다.

### 7-1. 일정
| 주차 | 작업 내용 |

| 1주 | 아이디어 확정, 프로토콜 명령어 초안 설계 |

| 3~4주 | 로그인 / 회원가입 / 개인 일정 CRUD |
| 5~6주 | 공유 캘린더 테이블·API·UI |

| 12주 | 테스트, 문서화, 발표 준비 |

### 7-2. 기술적 이슈와 해결

**① TCP 부분 수신 문제**
한 번의 `readyRead()` 에 여러 메시지가 합쳐 오거나, 반대로 한 메시지가 쪼개져 오는 현상 발견.
→ 소켓별 `QMap<QTcpSocket*, QString> m_buffers` 를 두고, `\n` 이 나타날 때까지 누적 후 분할 처리.

**② 프로필 사진 전송 시 `:` 충돌**
Base64 는 `+`, `/`, `=` 만 포함하지만 우리 프로토콜 구분자는 `:`. 안전하나, 방어적으로 **PROFILE_RES 의 마지막 필드는 `mid(2).join(":")`** 로 복원하여 미래 변경에도 안전하게 설계.

**③ 재접속 시 상태 꼬임**
서버 재시작 후 클라이언트가 자동 재접속하면, 온라인 목록·공유 캘린더 탭이 중복 생성되는 이슈.
→ `m_initialOnlineReceived`, `m_sharedCals` 클리어 후 재동기화 플로우 추가.

**④ 친구 목록 UX 개선 (후반 리팩터링)**
초기에는 단순 `QPushButton` 나열 → 후반에 **원형 프로필 사진 + 온라인 점 + 이름** 형태의 위젯 기반으로 재설계.

## 8. 결론 및 향후 개선 방향

### 8-1. 결론
기획 단계에서 설정한 "일정과 대화가 한 맥락에 묶인 협업 앱" 이라는 목표를 **모든 핵심 기능(개인/공유 일정, 3종 채팅, 친구 상태, 공유캘린더 내 채팅)** 에서 달성했습니다. 동시에 기획에 없었던 사용성 개선(다크모드, 트레이, 자동 재접속, 프로필)을 추가로 구현하여 실사용에 가까운 완성도를 확보했습니다.

### 8-2. 팀이 얻은 것
- Qt 시그널/슬롯, 위젯 커스터마이징, 멀티다이얼로그 UI 설계 경험
- **30개 이상의 명령어를 직접 설계·구현**하며 얻은 텍스트 기반 프로토콜 감각
- SQLite 기반 관계형 스키마 설계 (1:N, N:M 관계 포함)
- 비동기 TCP 통신·부분 수신·브로드캐스트 처리 경험

### 8-3. 향후 개선
- **검색 기능 추가** : 현재는 일정·채팅의 입력/수정/삭제 (CUD) 만 가능. 키워드 기반 일정 검색 → 채팅 메시지 검색 → 통합 검색(Spotlight 스타일) 순서로 확장 예정
- **보안** : 현재 비밀번호 평문 저장 → bcrypt / Argon2 해싱 필요
- **프로토콜** : 텍스트 → Protobuf / JSON 로 전환하여 확장성 확보
- **모바일 포팅** : Qt for Android / iOS, 또는 Flutter 재작성
- **푸시 알림** : 앱이 꺼져있어도 일정·멘션 알림
- **파일 전송** : 채팅에서 이미지·파일 공유
- **화상 회의** : 공유 캘린더 일정에서 바로 통화 시작

## 부록 A. 빌드 및 실행 방법

### 요구 사항
- Qt 6.11 이상 (Widgets, Network, Sql, Multimedia)
- CMake 3.16 이상
- C++17 지원 컴파일러 (MinGW 64-bit / MSVC)

### 빌드
```bash
mkdir build && cd build
cmake ..
cmake --build . --config Release
```

### 실행
1. `Server` 먼저 실행 (포트 25000 리스닝)
2. `Client` 실행 → 서버 IP 입력 → 로그인/회원가입

### 설정 파일
- `secrets.h.example` 를 복사해 `secrets.h` 생성 후, Google OAuth Client ID / 기상청 API 키 입력
